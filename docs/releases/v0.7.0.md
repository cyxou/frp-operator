Releasing v0.7.0 FRP Operator 🥳🥳

Changes:
1. Secure access to FRP Server with OIDC, TLS, and mutual TLS verification
2. Introducing HTTP & HTTPS Upstream support
3. Introducing FRP plugins, transport tuning, load balancing, and TCPMUX for Upstreams
4. Pod templates for the FRP client pod (resources, node selector, labels, tolerations, etc.)
5. Validation for duplicate Upstream server ports & Visitor ports
6. Reliable config reload using exec-based ConfigMap sync verification (replaces `sleep`-based waits)
7. New `enableAssistedAddrs` option on XTCP Visitors for fine-grained NAT-traversal control
8. Remove the deprecated `kube-rbac-proxy` sidecar — the operator now serves secure metrics natively on `:8443` with Kubernetes TokenReview/SubjectAccessReview

## How to Upgrade

Helm by default doesn't update the CRDs, you need to apply the CRDs manually:
```
kubectl apply -f https://raw.githubusercontent.com/zufardhiyaulhaq/frp-operator/v0.7.0/charts/frp-operator/crds/crds.yaml
```

⚠️ **Breaking change — metrics endpoint:** The `kube-rbac-proxy` sidecar is gone. Metrics are now served directly by the manager on `:8443`. Prometheus scrapers continue to work via the existing `*-controller-manager-metrics-service` Service if they authenticate with a ServiceAccount token bound to the `*-metrics-reader` ClusterRole. The `gcr.io/kubebuilder/kube-rbac-proxy` image is no longer pulled.
